## 항목별 검증 결과

### 1. 투명 윈도우 + TextBox (✅ 통과)
WPF AllowsTransparency=True + 표준 TextBox는 검증된 조합이다. TextBox의 Background을 Transparent로 설정하면 투명 배경 위 텍스트 표시가 자연스럽다.

### 2. 드래그 영역 크기 (⚠️ 주의)
Fitts' Law 기준 8px은 타겟이 너무 작다. 특히 투명 배경에서는 드래그 영역이 시각적으로 불분명하므로, 크기와 어포던스 모두 보완 필요.

**권고**: 12px + 마우스 오버 시 미세 하이라이트 (배경색 5% 불투명도). 투명 메모장 특성상 드래그 영역이 크면 메모 공간이 줄어들므로 12px이 균형점.

### 3. 리사이즈 핸들 크기 (⚠️ 주의)
6px 리사이즈 핸들은 정밀 조작이 필요하다. 8px 이상으로 확대하고, 마우스 커서를 ResizeNESW/ResizeNWSE 등으로 변경하여 어포던스를 제공해야 한다.

**권고**: WindowChrome의 ResizeBorderThickness를 사용하면 시스템이 리사이즈 영역을 관리하여 더 안정적이다.

### 4. 투명도 조절 방식 (⚠️ 주의)
WPF ContextMenu 내 Slider 커스텀 아이템은 구현 복잡도가 높고, 메뉴 닫힘/포커스 이슈가 발생할 수 있다.

**권고**: Phase 1(Sprint 2)에서는 단계별 선택(20%/40%/60%/80%/100%)이 안정적이다. Phase 3에서 ContextMenu 확장 시 슬라이더로 업그레이드 가능.

### 5. 키보드 흐름 (✅ 통과)
- AcceptsTab=True: Tab키가 텍스트 내 탭 입력으로 사용 -> 메모장에서 자연스러운 동작
- AcceptsReturn=True: Enter키가 줄바꿈 -> 정상
- Ctrl+A/C/V/X/Z/Y: TextBox 기본 지원 -> 추가 구현 불필요

### 6. 최소 윈도우 크기 (✅ 통과)
200x150은 텍스트 입력 영역으로 실용적인 최소 크기이다.

### 7. WindowChrome 활용 (⚠️ 주의)
커스텀 리사이즈 핸들보다 WindowChrome을 사용하면:
- ResizeBorderThickness로 리사이즈 영역을 시스템이 관리
- CaptionHeight=0 + 커스텀 드래그 영역으로 타이틀바 대체
- CornerRadius로 윈도우 모서리 처리
- 구현 복잡도가 낮아짐

**권고**: WindowChrome 채택을 강력 권고.

## 리스크 및 대안

1. **AllowsTransparency + 소프트웨어 렌더링**: 메모장 용도에서는 영향 미미하나, TextBox에 대량 텍스트(수천 줄) 입력 시 스크롤 성능 확인 필요. Phase 1 완료 후 성능 측정 권고.
2. **드래그 영역 시각적 피드백**: 투명 모드에서 드래그 영역이 보이지 않으면 사용자가 윈도우 이동 방법을 모름. IsMouseOver 트리거로 미세 하이라이트 추가.
3. **Phase 2 텍스트 테두리 전환**: Phase 2에서 TextBox -> 커스텀 컨트롤 전환이 필요하므로, TextBox를 ContentControl 등으로 감싸서 교체 용이한 구조 권고.
